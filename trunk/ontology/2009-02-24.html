This ontology defines terms for modelling semantic, interoperable gadgets or widgets. It has been developed as part of the EU project 'FAST' (FAST AND ADVANCED STORYBOARD TOOLS), FP7-ICT-2007-1-216048.

Overview

Classes

Class: Resource

LabelResource
super-class-offast:Screen_Flow, fast:Screen, fast:Backend_Service, fast:Flow_Control_Element, fast:Operator, fast:Form_Element, fast:Condition, fast:WithConditions
in-domain-offast:contains, fast:hasIcon, fast:hasScreenshot
in-range-offast:contains

Anything that is part of a gadget (or the gadget itself). Tentatively anything that can be 'touched' and moved around in the FAST IDE.


Class: Screen_Flow

LabelScreen Flow
sub-class-offast:Resource

The complete gadget, a set of screens.


Class: Screen

LabelScreen
sub-class-offast:Resource

An individual screen.


Class: Backend_Service

LabelBackend Service
sub-class-offast:Resource

A Web service which provides data and/or functionality to a screen. A backend service will often be external to FAST, and will probably have to be wrapped by the screen.


Class: Flow_Control_Element

LabelFlow Control Element
super-class-offast:Screen_Flow_Start, fast:Screen_Flow_End, fast:Connector
sub-class-offast:Resource

Any kind of component which can restrict the default flow of screens in a gadget.


Class: Screen_Flow_Start

LabelScreen Flow Start
sub-class-offast:Flow_Control_Element

The entry point to a wigdet; the first screen.


Class: Screen_Flow_End

LabelScreen Flow End
sub-class-offast:Flow_Control_Element

A screen that ends the workflow of the gadget.


Class: Connector

LabelConnector
sub-class-offast:Flow_Control_Element

An explicit connection between two screens.


Class: Operator

LabelOperator
sub-class-offast:Resource

Any kind of component that is used to connect backend services to form elements. Examples are simple pipes, aggregators or various kinds of filters.


Class: Form_Element

LabelForm Element
sub-class-offast:Resource

Form elements are UI elements in a particular screen.


Class: Condition

LabelCondition
sub-class-offast:Resource
in-domain-offast:hasPattern, fast:isPositive
in-range-offast:hasPreCondition, fast:hasPostCondition

The pre- or post-condition of either a screen or a screenflow. In the latter case, each target platform will use these conditions in its own way, or may also ignore them. E.g., in EzWeb pre- and post-conditions correspond to the concepts of slot and event.


Class: WithConditions

LabelWith-condition
sub-class-offast:Resource
union-offast:Screen, fast:Screen_Flow
in-domain-offast:hasPreCondition, fast:hasPostCondition

Those kinds of resource which can have pre- or post-conditions (i.e., screens and screen flows).

Many kinds of components in FAST can contain other components: screenflows contain screens, screens contain forms or form elements, etc.


Class: hasPreCondition

Labelhas pre-condition
Typeowl:ObjectProperty
domainfast:WithConditions
domainfast:Condition

This property links a screen or screenflow to its pre-condition, i.e., the facts that need to be fulfilled in order for this screen or screenflow to be reachable.


Class: hasPostCondition

Labelhas post-condition
Typeowl:ObjectProperty
domainfast:WithConditions
domainfast:Condition

This property links a screen or screenflow to its post-condition, i.e., the facts that are produced once the screen or screenflow has been executed.

Class: hasScreenshot

This property links a screen or screenflow to its pre-condition, i.e., the facts that need to be fulfilled in order for this screen or screenflow to be reachable.


Class: isPositive

Labelis positive
Typeowl:DatatypeProperty
domainfast:Condition
domainxsd:boolean

Conditions can be positive or negative, depending on whether they must be fulfilled or must not be fulfilled (in the case of pre-conditions), or whether their facts will be added to the canvas or removed (in the case of post-conditions).
